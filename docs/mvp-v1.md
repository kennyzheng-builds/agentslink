# Agent Link MVP v1：协作上下文包

> 状态：方案已确认，待开发
> 更新时间：2026-03-08
> 核心思路：Agent 打包问题上下文 → 人传递 → 对方 Agent 读取分析 → 打包回复 → 人传回

---

## 核心问题

人类在两个 Agent 之间传话，会造成严重的信息丢失。技术问题中的报错信息、代码片段、配置细节、已尝试方案，经过人工转述后大量丢失。但如果 Agent 有完整上下文，大多数问题它是能解决的。

## 核心思路

**不需要两个 Agent 实时对话，只需要让 Agent（而非人类）来打包和解读上下文。**

人类依然负责传递（微信发个文本/链接），但打包和解读都由 Agent 完成，消除信息损耗。

---

## 用户流程

```
Kenny 遇到飞书权限问题，自己的 Agent 解决不了
    ↓
Kenny："帮我把这个问题打包，我找老王帮忙看看"
    ↓
Agent 自动收集当前上下文，生成「协作请求」（结构化 markdown）
    ↓
Kenny 复制这段文本，微信发给老王
    ↓
老王把文本粘贴给自己的 Agent
    ↓
老王的 Agent 自动识别这是「协作请求」，读取完整上下文
    ↓
Agent 分析问题，生成「协作回复」
    ↓
老王复制回复文本，微信发回给 Kenny
    ↓
Kenny 粘贴给自己的 Agent
    ↓
Agent 读取回复，整合建议，告诉 Kenny 下一步怎么做
```

### 可见性

全程在主人和 Agent 的对话流中完成，主人能直接看到：
- Agent 打包了什么内容（"我帮你整理了以下问题描述..."）
- 收到的回复内容（"对方 Agent 的分析结果是..."）
- Agent 基于回复的行动建议

不需要额外的"子 Agent 面板"或"协作控制台"。

---

## 协作模式：只读咨询

v1 只做**只读咨询**——发问题、收建议，不涉及写操作或远程执行。

| 允许 | 不允许 |
|------|--------|
| 文本问答 | 写文件 |
| 只读分析 | 执行命令 |
| 返回建议/方案 | 调用外部工具 |
| 提供操作步骤 | 访问对方本地文件 |

这样做的好处：
- 安全边界清晰（只读 = 无需复杂权限控制）
- 信任门槛低（朋友只是帮你看一下，不碰你的环境）
- 解决了核心痛点（上下文无损传递）
- 开发量极小

## 协作请求格式

Agent 生成的「协作请求」是一段结构化 markdown：

```markdown
<!-- AGENT-LINK-REQUEST v1 -->

# 协作请求：飞书权限问题

**来自**：Kenny 的 Agent
**时间**：2026-03-08 14:30
**类型**：bug 排查

## 问题描述
在调用飞书开放平台 API 获取通讯录时，返回 permission denied 错误...

## 环境信息
- 飞书应用类型：企业自建应用
- SDK 版本：feishu-sdk@2.3.1
- Node.js 版本：20.11.0

## 报错信息
```
Error: Request failed with status code 403
Response: {"code":10003,"msg":"permission denied: ..."}
```

## 已尝试方案
1. ✅ 检查了应用权限配置 → 权限列表中已添加「通讯录读取」
2. ✅ 重新生成了 App Token → 问题依旧
3. ❌ 未检查租户管理员审批状态

## 期望
定位 403 错误的根因，给出解决步骤。

<!-- END AGENT-LINK-REQUEST -->
```

### 格式设计原则

- **HTML 注释标记**：`<!-- AGENT-LINK-REQUEST v1 -->` 让 Agent 能自动识别，人类阅读时不受影响
- **纯 markdown**：不依赖任何特殊格式或工具，微信/飞书/邮件都能发
- **结构化分节**：确保对方 Agent 能精准提取每个部分
- **人类可读**：即使不用 Agent，人直接看也能理解

## 协作回复格式

对方 Agent 生成的「协作回复」：

```markdown
<!-- AGENT-LINK-RESPONSE v1 -->

# 协作回复：飞书权限问题

**来自**：老王的 Agent
**时间**：2026-03-08 14:45
**针对**：Kenny 的 Agent 的协作请求

## 诊断结果
错误码 10003 表示应用虽然在权限列表中添加了「通讯录读取」，但**租户管理员尚未审批该权限**...

## 建议方案

### 方案 A（推荐）：申请管理员审批
1. 登录飞书管理后台 → 应用管理 → 找到你的应用
2. 查看「权限管理」→ 「待审批权限」
3. 联系租户管理员点击「审批通过」
4. 等待 5-10 分钟生效后重试

### 方案 B：使用 User Token 替代
如果无法获得管理员审批...

## 补充说明
- 飞书权限分为「应用自身权限」和「需管理员审批权限」，通讯录属于后者
- 建议同时检查 IP 白名单配置

## 参考文档
- 飞书开放平台权限说明：https://...

<!-- END AGENT-LINK-RESPONSE -->
```

## 追问机制

如果一轮不够，支持追问：

```markdown
<!-- AGENT-LINK-FOLLOWUP v1 -->

# 追问：飞书权限问题

**来自**：Kenny 的 Agent
**时间**：2026-03-08 15:00
**上下文**：基于老王 Agent 的回复

## 追问内容
按照方案 A 操作后，管理员已审批，但 API 调用仍返回 10003。
新增信息：应用的 IP 白名单为空（未配置）。

## 新增报错
```
Same error after admin approval, checked IP whitelist: empty
```

<!-- END AGENT-LINK-FOLLOWUP -->
```

人工传递追问文本即可，流程与首次一致。

## 技术实现

### 唯一交付物：一个 OpenClaw Skill

```
skills/agent-link/
├── SKILL.md          # Skill 描述文件（prompt + 能力声明）
└── (无其他文件)
```

**零依赖、零基础设施、零成本。**

Skill 的全部能力通过 SKILL.md 中的 prompt 定义，不需要额外代码。

### SKILL.md 需要定义的能力

#### 1. 打包能力（生成协作请求）

触发方式：
- "帮我打包这个问题"
- "我要找人帮忙看看"
- "生成协作请求"

Agent 行为：
1. 从当前对话上下文中提取：问题描述、环境信息、报错信息、已尝试方案
2. 整理成 `AGENT-LINK-REQUEST v1` 格式
3. 输出给主人，提示"复制这段内容发给你的朋友"

#### 2. 识别 + 分析能力（处理协作请求）

触发方式：
- 用户粘贴了包含 `<!-- AGENT-LINK-REQUEST v1 -->` 标记的文本
- "帮我看看这个问题"

Agent 行为：
1. 识别标记，解析结构化内容
2. 基于自身能力分析问题
3. 生成 `AGENT-LINK-RESPONSE v1` 格式的回复
4. 输出给主人，提示"复制这段回复发回给对方"

#### 3. 回复解读能力（处理协作回复）

触发方式：
- 用户粘贴了包含 `<!-- AGENT-LINK-RESPONSE v1 -->` 标记的文本

Agent 行为：
1. 识别标记，解析回复内容
2. 结合之前的问题上下文，整合建议
3. 告诉主人下一步怎么做，哪些需要主人手动操作

#### 4. 追问能力

触发方式：
- "还有问题，帮我继续追问"
- "方案 A 试了不行"

Agent 行为：
1. 结合之前的请求和回复，生成 `AGENT-LINK-FOLLOWUP v1`
2. 输出给主人传递

### 展示名

首次使用时 Agent 询问："你希望在协作中怎么称呼？"
保存到本地配置 `~/.agent-link/config.json`，之后所有请求/回复中显示为"XX 的 Agent"。

兜底链：环境变量 `AGENT_LINK_DISPLAY_NAME` → 系统用户名 → "匿名 Agent"。

## 安全设计

### 只读模式天然安全

因为 v1 只做只读咨询，安全边界非常简单：
- 不涉及远程执行 → 无需权限控制
- 不涉及文件传输 → 无需沙箱
- 信息由人手动传递 → 人自然会过滤不想发的内容

### 打包时的敏感信息过滤

Agent 在打包时应自动过滤：

| 类型 | 处理 |
|------|------|
| API Key / Token（`sk-`, `ghp_` 等） | 替换为 `[API_KEY_REDACTED]` |
| 密码字段（`password=xxx`） | 替换为 `[PASSWORD_REDACTED]` |
| 私钥 | 不包含在请求中 |
| 本地绝对路径 | 替换为相对路径 |

这些规则写在 SKILL.md 的 prompt 中，由 Agent 在打包时执行。

### Prompt 级安全指令

SKILL.md 中明确告知 Agent：
1. 打包时不要包含凭证、密码、私钥
2. 如果解决问题需要凭证，只说"需要 XX 类型的凭证"
3. 报错信息中的用户名、内部 IP 等做脱敏处理

## 开发计划

| 阶段 | 目标 | 时间 |
|------|------|------|
| **Week 1** | SKILL.md 编写 + 打包/识别/回复逻辑 | 3-5 天 |
| **Week 1** | 本地测试（两个 OpenClaw 实例互发） | 1-2 天 |
| **Week 2** | 真人测试（找朋友实际使用） | 持续 |
| **Week 2** | 根据反馈迭代格式和 prompt | 持续 |

### 验证指标

| 指标 | 目标 |
|------|------|
| 协作完成率 | > 60%（对方能给出有用回复） |
| 信息完整性 | 对方 Agent 无需追问即可分析 |
| 用户体验 | 比"人工复制粘贴聊天记录"明显更好 |
| 再次使用率 | 用过的人愿意第二次使用 |

## v1 → v2 演进

v1 验证成功后，v2 加入实时 Agent-to-Agent 通信：

| 维度 | v1（当前） | v2（未来） |
|------|-----------|-----------|
| 传递方式 | 人工复制粘贴 | Agent 通过 A2A 协议直连 |
| 交互模式 | 异步（请求-回复） | 实时多轮对话 |
| 基础设施 | 无（纯 Skill） | A2A Gateway + 服务端 |
| 平台 | OpenClaw 单平台 | OpenClaw + Claude Code + 更多 |
| 成本 | $0 | 域名 + CF Workers |
| 协作模式 | 只读咨询 | 只读 + 受限执行 |

v1 的核心资产（格式规范、打包逻辑、prompt 设计）可直接复用到 v2。
