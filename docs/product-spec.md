# AgentsLink 产品方案

> 更新时间：2026-03-08

## 一句话价值

让 AI Agent 之间无损传递上下文来协作解决问题，人类只需转发、确认关键决策。

---

## 要解决的问题

你和朋友各有一个 Agent。你遇到技术问题，朋友的 Agent 擅长解决。

**现在的做法**：你描述问题 → 朋友转述给他的 Agent → Agent 回答 → 朋友转述给你 → 你喂给你的 Agent。每一手都在丢信息，尤其是代码、报错、配置这些精确细节。

**AgentsLink 的做法**：你的 Agent 把问题打包成结构化上下文 → 你转发给朋友 → 朋友的 Agent 读取完整上下文，直接分析 → 回复打包传回。**信息零损耗。**

核心洞察：Agent 只要有足够的上下文，就能解决问题。瓶颈不在 Agent 的能力，而在人类传话造成的信息丢失。

---

## 目标用户

高频跨人协作的技术用户，特别是：
- 每天都在 OpenClaw / Claude Code / 本地 coding agent 里工作的开发者
- 强关系协作网络中的求助场景（朋友、同事、小团队）
- AI Power Users（手里不止一个 Agent，乐于探索新工具）

**不是**通用 Agent 社交网络。是高信任关系上的协作工具。

---

## 用户场景

### 场景 1：技术问题求助（核心场景）

```
Kenny 的 Agent 遇到飞书权限问题
    ↓
Kenny："帮我把这个问题打包，我找老王帮忙看看"
    ↓
Agent 收集上下文（报错、代码、已尝试方案），生成协作请求
    ↓
Kenny 复制发给老王（微信/飞书）
    ↓
老王粘贴给自己的 Agent → Agent 识别、分析、生成回复
    ↓
老王把回复发回给 Kenny → Kenny 的 Agent 读取并应用建议
    ↓
问题解决
```

### 场景 2：代码 Review 求助

```
开发者 A 遇到性能问题，自己的 Agent 没有思路
    ↓
Agent 打包：代码片段 + 性能指标 + 已尝试优化
    ↓
发给擅长性能优化的朋友 → 朋友的 Agent 分析并给出建议
```

## 协作模型：老板-下属

```
老板 A ←────人传递上下文包────→ 老板 B
   ↕                             ↕
  指示打包/读取回复             指示分析/打包回复
   ↕                             ↕
下属 a（打包/解读）            下属 b（分析/回复）
```

**老板（人类）的控制**：
- 决定找谁帮忙（基于对人的信任）
- 看到 Agent 打包的完整内容（在对话流中直接可见）
- 手动传递（天然过滤不想发的信息）
- 看到对方回复并决定是否采纳

**下属（Agent）的职责**：
- 高质量打包问题上下文
- 自动识别并解析收到的协作请求
- 基于完整上下文分析问题
- 把回复整合成可执行的建议

## 产品演进

### v1：异步上下文包（当前）

- **模式**：Agent 打包 → 人传递 → Agent 分析回复
- **平台**：仅 OpenClaw
- **协作类型**：只读咨询（发问题、收建议）
- **基础设施**：无（纯 Skill 文件）
- **成本**：$0

详见 [MVP v1 方案](./mvp-v1.md)

## 身份与展示

### 展示名

- 首次使用时 Agent 主动问："你希望在协作中怎么称呼？"
- 保存到 `~/.agents-link/config.json`
- 协作请求/回复中显示为"XX 的 Agent"
- 兜底：环境变量 → 系统用户名 → "匿名 Agent"

### 隐私

- v1 由人手动传递，天然安全
- Agent 打包时自动过滤 API Key、密码等敏感信息
- 不存在服务端 → 不存在数据泄露风险
