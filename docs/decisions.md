# Agent Link 决策记录

## 核心原则

### 为什么做这个项目？

> "我的 Agent 有一个问题解决不了，只能问其他人，其他人再问他的 Agent，人类在中间传达，效率很低，而且人类的文字容易信息折损。"

**核心价值**：消除人类在 Agent 之间传话造成的信息损耗。Agent 只要有足够的上下文，就能解决问题。

### 安全模型

> "在目前的 Agent 早期，人来保证 Agent 的安全，以及知道哪个 Agent 能帮你解决。"

- **人工牵线**：人类决定找谁帮忙（基于对人的信任）
- **人类传递**（v1）：人手动传递上下文包，天然控制信息流
- **Agent 自主对话**（v2）：建立连接后 Agent 直接交流
- **人类确认收尾**：关键决策点需要主人批准

## 关键决策

### 1. 协作模式：老板-下属模型

```
老板 A ←────确认合作────→ 老板 B
   ↓                       ↓
下属 a ←────具体执行────→ 下属 b
```

- 老板拍板：建联、关键决策、关闭会话
- 下属执行：技术细节沟通、方案尝试

### 2. MVP 分阶段：先异步再实时（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| v1 直接做实时 A2A 通信 vs 先做异步上下文包 | **先做异步上下文包** | 零成本验证核心假设 |

**关键洞察（来自 Gemini + ChatGPT Review）**：

真正要解决的是"上下文无损传递"，而不是"两个 Agent 实时聊天"。只读咨询场景下，一个完整的上下文包 + 一个完整的分析回复就够了。

- **v1**：Agent 打包上下文 → 人传递 → 对方 Agent 分析回复（零基础设施）
- **v2**：Agent 通过 A2A 协议实时对话（需要 Gateway 和服务端）

### 3. v1 只做只读咨询模式（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| 通用协作 vs 只读咨询 | **只读咨询** | 安全边界清晰，信任门槛低，解决核心痛点 |

- 允许：文本问答、只读分析、返回建议
- 不允许：写文件、执行命令、调外部工具
- ChatGPT 的建议："不是让两个 agent 自由合作，而是可控委托"

### 4. v1 只做 OpenClaw 单平台（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| 双平台首发 vs 单平台 | **OpenClaw 单平台** | 聚焦精力，OpenClaw 正火用户多 |

- 平台适配层"很薄"大概率不成立（ChatGPT 指出）
- 先在一个平台上打透，再扩展到 Claude Code
- Claude Code 支持放到 v2

### 5. 通信协议：采用 Google A2A（v2 使用）

选择 A2A 作为 v2 实时通信的核心协议。

**为什么选 A2A**：
- 2025 年发布后 50+ 企业参与，捐给 Linux 基金会
- 跨平台 Agent 互操作是其设计目标
- Task + input-required 天然支持人类介入

**注意事项（ChatGPT 指出）**：
- 协议互通 ≠ 语义互通，需要在 A2A 之上定义更窄的 interaction contract
- Gateway 不只是"中转站"，实际是产品控制面
- "不存储对话内容"在中转架构下需要更严谨的表述

### 6. 零成本策略（2026-03-08 决策）

个人项目，不能有固定成本。

| 阶段 | 成本 | 说明 |
|-----|------|------|
| v1 | $0/月 | 纯 Skill 文件，无服务端 |
| v2 | 域名 ~$10/年 + CF Workers 免费额度 | 尽量控制在最低 |

### 7. 敏感信息安全

**v1 方案**：Prompt 级安全
- Agent 打包时自动过滤 API Key、密码、私钥
- 规则写在 SKILL.md prompt 中
- 人手动传递天然提供一层安全（人会过滤不想发的内容）

**v2 方案**：Prompt + 正则 + 结构化 DLP
- 消息发送管道中加入正则匹配
- ChatGPT 建议的多层安全模型（能力最小权限、结构化分类、双边 consent、可审计性）

### 8. 展示名机制

首次使用时 Agent 主动问主人："你希望在协作中怎么称呼？"
- 保存到 `~/.agent-link/config.json`
- 兜底：环境变量 → 系统用户名 → "匿名 Agent"

## 外部 Review 摘要

### Gemini (gemini-2.5-pro) 主要意见

- 产品方向正确，建议立即启动
- 协作总结应提到 P0（信任需要可见性）
- 加入 TTL / 消息次数限制防止 Agent 死循环
- 链接加入 Nonce/Token 防止身份伪造
- Gateway 需要 Heartbeat 机制

### ChatGPT (o3) 主要意见

- 问题成立但首战边界应该更窄
- "协议互通 ≠ 语义互通"是最大隐含风险
- "为什么朋友愿意帮你"这个激励问题没解决
- 正则过滤远远不够，需要多层安全模型
- "Gateway 不存储内容"说法过度承诺
- 建议先做"受限协作"而非"自由协作"
- 产品边界不清（同时像协议网关、协作产品、身份系统、安全中间层）

**综合结论**：采纳 ChatGPT 的"缩首战边界"建议，v1 做异步上下文包 + 只读咨询 + 单平台。

详细 review 见 [reviews/](./reviews/) 目录。

## 术语定义

| 术语 | 含义 |
|-----|------|
| **主人** | 拥有 Agent 的人类用户 |
| **Agent** | 主人的 AI 助手（OpenClaw / Claude Code 等） |
| **协作请求** | Agent 打包的结构化问题上下文（v1 核心概念） |
| **协作回复** | 对方 Agent 的结构化分析回复 |
| **A2A** | Google Agent-to-Agent 协议（v2 使用） |
| **Gateway** | A2A 网关，帮本地 Agent 做中转（v2 使用） |
