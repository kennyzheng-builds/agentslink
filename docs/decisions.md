# AgentsLink 决策记录

## 核心原则

### 为什么做这个项目？

> "我的 Agent 有一个问题解决不了，只能问其他人，其他人再问他的 Agent，人类在中间传达，效率很低，而且人类的文字容易信息折损。"

**核心价值**：消除人类在 Agent 之间传话造成的信息损耗。Agent 只要有足够的上下文，就能解决问题。

### 安全模型

> "在目前的 Agent 早期，人来保证 Agent 的安全，以及知道哪个 Agent 能帮你解决。"

- **人工牵线**：人类决定找谁帮忙（基于对人的信任）
- **人类传递**（v1）：人手动传递上下文包，天然控制信息流
- **人类确认收尾**：关键决策点需要主人批准

## 关键决策

### 1. 协作模式：老板-下属模型

```
老板 A ←────确认合作────→ 老板 B
   ↓                       ↓
下属 a ←────具体执行────→ 下属 b
```

- 老板拍板：建联、关键决策、关闭会话
- 下属执行：技术细节沟通、方案尝试

### 2. MVP 分阶段：先异步再实时（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| 实时通信 vs 异步上下文包 | **异步上下文包** | 零成本验证核心假设 |

**关键洞察**：

真正要解决的是"上下文无损传递"，而不是"两个 Agent 实时聊天"。只读咨询场景下，一个完整的上下文包 + 一个完整的分析回复就够了。

Agent 打包上下文 → 人传递 → 对方 Agent 分析回复，零基础设施。

### 3. v1 只做只读咨询模式（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| 通用协作 vs 只读咨询 | **只读咨询** | 安全边界清晰，信任门槛低，解决核心痛点 |

- 允许：文本问答、只读分析、返回建议
- 不允许：写文件、执行命令、调外部工具
- ChatGPT 的建议："不是让两个 agent 自由合作，而是可控委托"

### 4. v1 只做 OpenClaw 单平台（2026-03-08 决策）

| 方案 | 决策 | 理由 |
|-----|------|------|
| 双平台首发 vs 单平台 | **OpenClaw 单平台** | 聚焦精力，OpenClaw 正火用户多 |

- 平台适配层"很薄"大概率不成立（ChatGPT 指出）
- 先在一个平台上打透，再扩展到 Claude Code
- 后续可扩展到 Claude Code 等其他平台

### 5. 零成本策略（2026-03-08 决策）

个人项目，不能有固定成本。

| 阶段 | 成本 | 说明 |
|-----|------|------|
| v1 | $0/月 | 纯 Skill 文件，无服务端 |

### 6. 敏感信息安全

**v1 方案**：Prompt 级安全
- Agent 打包时自动过滤 API Key、密码、私钥
- 规则写在 SKILL.md prompt 中
- 人手动传递天然提供一层安全（人会过滤不想发的内容）

### 7. 展示名机制

首次使用时 Agent 主动问主人："你希望在协作中怎么称呼？"
- 保存到 `~/.agents-link/config.json`
- 兜底：环境变量 → 系统用户名 → "匿名 Agent"

## 外部 Review 摘要

### 外部 Review 综合结论

经过多方评审，核心共识：
- 问题成立，产品方向正确
- 首战边界应该更窄——先做"受限协作"而非"自由协作"
- 先验证核心假设（上下文无损传递），再扩展能力

**最终决策**：v1 做异步上下文包 + 只读咨询 + 单平台。

## 术语定义

| 术语 | 含义 |
|-----|------|
| **主人** | 拥有 Agent 的人类用户 |
| **Agent** | 主人的 AI 助手（OpenClaw / Claude Code 等） |
| **协作请求** | Agent 打包的结构化问题上下文（v1 核心概念） |
| **协作回复** | 对方 Agent 的结构化分析回复 |
