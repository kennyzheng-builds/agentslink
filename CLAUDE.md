# AgentsLink 项目规则

## 用户角色

用户是非技术背景的独立产品负责人（个人开发者），专注于业务和产品决策。所有开发流程的判断由 Claude 主动承担。资源有限，技术方案必须优先考虑低成本：优先免费/开源方案，避免高额账单，选型时主动说明成本。

## 工作流

### 整体流程

1. **讨论阶段** — 一个窗口内完成产品方案和技术方案的对齐
2. **开发阶段** — Claude 主动告知可以并行的任务，用户根据指引开多个窗口
3. **持续推进** — 每个窗口专注一个任务，上下文满时 clear 后说"继续"即可
4. **依赖处理** — Claude 自行起 sub-agent 处理独立子任务；需要用户决策时才请用户开新窗口

### 上下文恢复机制（关键）

**每次新对话或 clear 后，Claude 的第一个动作**必须是：
1. 读取 `docs/status.md`（项目总览和任务状态）
2. 读取对应任务文件 `docs/tasks/task-*.md`（具体进度）
3. 根据用户说的话（如"继续"、"继续做A"）自动恢复上下文，无需用户重复解释

### 任务状态维护

Claude 必须在以下时机主动更新 `docs/status.md` 和对应任务文件：
- 任务开始时：创建任务文件，更新 status.md
- 阶段性进展时：记录完成了什么、下一步是什么
- 遇到阻塞/依赖时：记录阻塞原因和待决事项
- 任务完成时：标记完成，更新 status.md
- clear 之前：确保当前进度已写入文件

## 技术决策原则

1. **调研先于实现** — 每个技术需求先调研现有方案（开源库、SaaS、API），确认没有合适的现成方案后再自己实现。调研可用 sub-agent 并行执行，不阻塞主流程。
2. **提前暴露依赖** — 尽早识别需要用户配合的事项（注册账号、申请 API key、开通服务等），提前告知用户准备，不要等到被卡住才说。用户是 Claude 在物理世界的执行者。

## 核心规则：主动引导开发流程

在整个项目开发过程中，Claude 必须主动在以下时机给出建议，而不是等用户来问：

1. **何时 commit** — 当一组有意义的改动完成后，主动提醒用户保存进度
2. **何时多开窗口** — 当存在可并行推进的独立任务时，主动建议用户开几个窗口分别做什么
3. **何时 clear** — 当对话过长影响效率时，主动建议 clear 并确保进度已保存到文件
4. **何时将讨论写入文件** — 当对话中的决策或内容足够成熟，应该持久化到项目文档时，主动执行
5. **目录结构管理** — 主动规划和调整项目的文件夹结构，保持清晰有序
6. **Git 工作流** — 分支、合并、版本管理等所有 git 操作由 Claude 判断和执行
7. **依赖与阻塞** — 发现当前任务依赖其他工作时，主动建议开新窗口或自行起 agent 处理

## 反馈自动持久化

当用户对协作方式给出反馈，Claude 必须：
1. 立刻在当次对话中调整行为
2. 提炼反馈背后的底层原则（而非记录具体细节）
3. 更新 CLAUDE.md 中对应的原则表述，保持规则精简、抽象、不堆砌

CLAUDE.md 只放原则，不放琐碎细节。定期审视并合并冗余规则。

## 沟通语言

使用中文与用户沟通。
